## Project Overview

**SpotMe** is an adaptive gym training companion built on **Java 21** with **Spring Boot 3.4** and **Domain-Driven Design (DDD)** principles. The backend uses gRPC as the primary transport, with future REST support. It dynamically adjusts workouts based on user feedback (RPE, DOMS, sleep quality) via a rules-based progression engine.

**Primary Transport:** gRPC on port 9090 (see `application.yaml`)  
**Database:** PostgreSQL with Flyway migrations  
**Architecture:** Hexagonal (ports & adapters) with strict layering

## Architecture & Module Organization

### Critical Pattern: Dependency Flow

The codebase enforces **strict unidirectional dependencies**:

```
domain/ (pure Java, zero frameworks)
    â†‘
application/ (use cases, depends only on domain)
    â†‘
adapters/ (in.grpc, out.persistence, out.rules, out.cache)
    â†‘
app/ (Spring Boot assembly, wires all modules)
```

**Why this matters:** Domain logic is testable without Spring. Adapters are replaceable. Never import `com.spotme.adapters.*` into domain or application.

### Key Modules

| Module | Purpose | Key Constraint |
|--------|---------|-----------------|
| **domain/** | Core business logic, aggregates, value objects, ports (interfaces) | NO Spring, NO framework dependencies beyond Jackson for JSON |
| **application/** | Use cases that orchestrate domain logic | Depends ONLY on domain; no Spring or adapters |
| **adapters/in.grpc** | gRPC service stubs (inbound) | Implements `application` use cases |
| **adapters/out.persistence** | JPA entities, repositories | Uses MapStruct for domainâ†”entity mapping |
| **adapters/out.rules** | Loads progression policy JSON from classpath (`/rules/*.json`) | Implements `RulesConfigPort` |
| **adapters/out.cache** | Redis session/caching (Redis starter added, not yet wired) | Optional; TestContainers in test config |
| **proto/** | Protobuf stubs (code-gen) | Auto-generated by Maven; do not edit |
| **app/** | Spring Boot entry point (`SpotMeApplication.java`) | Orchestrates all modules; test uses Testcontainers for Postgres |

## Essential Concepts & Patterns

### 1. **Strongly-Typed Identifiers**

All domain entities use dedicated ID classes (not raw strings/UUIDs). Examples:
- `UserId`, `ProgramId`, `ExerciseId`, `ExerciseDefinitionId`, `WorkoutSessionId`

**When adding a new aggregate:** Create a corresponding `*Id` value object in `domain/model/{context}/`.

### 2. **Domain Ports (Interfaces)**

Ports define boundaries between domain and infrastructure. Located in `domain/port/`:
- `RulesConfigPort` â†’ Loads progression rules JSON
- `WorkoutReadPort` / `WorkoutWritePort` â†’ Persist/retrieve workouts
- `ExerciseRepository` â†’ Exercise definitions (in-memory or DB)

**Pattern:** Domain defines the interface; adapters implement it. Do not let adapters dictate domain contracts.

### 3. **Progression Engine & Feedback Loop**

The core algorithm lives in `domain/rules/ProgressionEngine.java`:

```java
public Prescription nextFor(ExerciseId exerciseId, ProgressionInput input, ProgressionPolicy policy) {
    // Uses DOMS, RPE, last weight/reps to compute next prescription
    // Microload: +0.25kg unless severe DOMS â†’ mirror
}
```

**Key inputs (from `ProgressionInput`):**
- `lastTopSetWeightKg()` / `lastTopSetReps()`
- `doms()` (soreness level)
- Threshold from `ProgressionPolicy`

**Related:** `domain/feedback/FeedbackEngine.java` (interface) + `SimpleFeedbackEvaluationService` (impl).

### 4. **MapStruct Mappers (Adapter Pattern)**

Entityâ†”Domain mapping uses MapStruct in `adapters/out.persistence/`:
- Keeps domain models free of JPA annotations
- Processor auto-generates implementations at compile time
- Property mismatches cause **compile-time errors** (preferable to runtime surprises)

**Convention:** `ProgramMapper` maps `ProgramEntity` â†” `Program` (domain).

### 5. **Rules as JSON Configuration**

Progression policies live in `src/main/resources/rules/{version}.json`. Example structure:
```json
{
  "version": "1.0",
  "microLoadKg": 0.25,
  "severeDomsThreshold": 8
}
```

Loaded via `ClasspathRulesConfigAdapter` (implements `RulesConfigPort`). Allows swapping policies without code changes.

## Critical Workflows

### Build & Run

```bash
# Build all modules (from project root)
mvn clean install

# Run Spring Boot application
mvn spring-boot:run -pl app

# gRPC server starts on port 9090
# Health check: http://localhost:8080/actuator/health (if REST enabled)
```

### Tests

```bash
# Run all tests (includes integration tests with Testcontainers/Postgres)
mvn test

# Domain tests run WITHOUT Spring (fast)
mvn test -pl domain

# Integration tests in app/ use @SpringBootTest + @ActiveProfiles("test")
# Testcontainers auto-configures ephemeral Postgres (see TestcontainersConfiguration.java)
```

### Proto Compilation

```bash
# Auto-triggered by Maven; generates stubs in proto/target/generated-sources/
# Do not edit generated files; modify .proto source instead
mvn clean compile -pl proto
```

### Code Generation (MapStruct)

```bash
# Auto-triggered; generated mappers in target/generated-sources/annotations/
mvn clean compile -pl adapters/out.persistence
```

## Common Development Patterns & Gotchas

### âś… DO:
- **Keep domain pure.** Use `java.util.List` (not Stream API at boundaries for immutability).
- **Throw domain exceptions** from domain logic; let adapters handle translation to gRPC/HTTP status codes.
- **Use constructor-based immutability** in value objects (no setters).
- **Test domain logic in isolation.** Most domain tests should have zero Spring overhead.
- **Inject ports via constructors** in adapters; let Spring manage the graph.

### âťŚ DON'T:
- Import `org.springframework.*` in `domain/` or `application/`.
- Return JPA entities from repositories; always map to domain.
- Put business logic in gRPC service handlers; call application use cases.
- Use static initializers for configuration; wire via Spring Beans.

### đź”Ť Common Patterns:

**Adding a new use case:**
1. Create `UseCase` interface + implementation in `application/{context}/`
2. Define required ports (if new) in `domain/port/`
3. Wire Spring Bean in `app/config/` (or auto-scan if `@Component`)
4. Implement gRPC handler in `adapters/in.grpc/` calling the use case

**Adding a new entity/aggregate:**
1. Create domain model in `domain/model/{context}/`
2. Create JPA entity in `adapters/out.persistence/jpa/entities/`
3. Add MapStruct mapper in `adapters/out.persistence/jpa/mappers/`
4. Update repo interface in domain; implement Spring Data JPA in adapter
5. Test domain behavior **without** DB; integration tests verify mapping

## Debugging & IDE Setup

- **Scan base packages:** Spring is configured to scan `com.spotme` only (prevents auto-wiring of unrelated libraries).
- **Test profiles:** Use `@ActiveProfiles("test")` to override configuration; Testcontainers auto-wire DB connection.
- **gRPC breakpoints:** Debugger works normally; set breakpoints in service handlers in `adapters/in.grpc/`.
- **Proto regeneration:** If `.proto` files change, `mvn clean compile` regenerates stubs; refresh IDE if classes not found.
